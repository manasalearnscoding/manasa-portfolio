Case study

Market Risk IT Intern

Nomura Securities International
TL;DR: Shipped a proof-of-concept What-If tool pipeline improvement by integrating Python APIs into a Java OLAP system — wrote the spec and built the solution in 10 weeks.
My role
IT Intern
Timeline
10 weeks · Summer 2025
Team
Market Risk IT
Platform
Java Spring · Python · OLAP

The problem

The Market Risk IT team needed a way for analysts to run hypothetical scenario analysis, like "What if the S&P fell by 5%" to understand and analyze market risk. To do this, data is stored in and queried from OLAP cubes.

Querying these cubes requires performing a REST API call to fetch the data from the buckets they are stored in, which is slow. The platform that the team used to query the cube came out with a pythonic API for the Java based cube. My 10-week internship was the window to study the platform, the documentation, and the api, and create a proof-of-concept for the new API integration.

My role

I built the proof-of-concept implementation: a Python microservice exposing a REST API that the Java Spring application could call to execute scenario calculations and return structured results to the OLAP layer.

The team provided domain context on market risk calculations and reviewed my spec, the architecture decisions, the implementation, and the demo. My manager and I scoped the project together at week one, and I presented the working PoC at week ten.

How I worked through it

I made a timeline for my project which I roughly followed - I used the first two weeks to learn, the middle six to build-and-validate, and the final two to handoff my work.

1

System discovery

I spent the first week reading about the system the company used and the existing query code.

2

Technical specification

I drew out the new system to cover how the Java service would spawn and communicate with the Python process.

5

Demo, handoff & documentation

I presented the working PoC to the Market Risk IT lead and two senior analysts in week ten. I also wrote a handoff doc covering architecture decisions, known limitations, and a recommended path from PoC to production — so the next engineer wouldn't have to rediscover what I'd learned.

What I learned

This was my first internship since I joined college and I learned so much.

What worked

I had never heard about OLAP cubes before this project. I took a lot of time to learn and study the system even beyond work hours and had a ton of support from my mentor and manager who answered all my questions. I even reached out to teams working on different parts of the system to get a comprehensive understanding of what I was working on. I am glad I utilised all the resources at my disposal.

What I'd change

I would have set up the staging test environment sooner. Waiting until mid-project gave me less time to work with inevitable issues like issues with downloading massive amounts of staging data.

Unexpected challenge

Legacy systems are massive and hard to understand as they are often not well documented. Asking the right questions and narrowing my scope for focus really saved me time and effort.

Skill gained

Learning about, demoing, and answering questions about a product feature in a short time span in a room full of senior engineers.
